01 — Contexte

Le contexte

  • Projet de 2e annee avec Adil : plateforme de RDV en Python/Flask
  • Fonctionnel en local mais jamais deploye, UI basique, code monolithique
  • Objectif V2 : reprendre le projet, le migrer vers une stack moderne et le mettre en production
  • Apprendre un workflow d'equipe pro : Git Flow, code review, CI/CD, deploiement cloud

Migration Python/Flask → Next.js + TypeScript : meme idee, stack plus moderne, plus scalable, avec un vrai workflow de developpement professionnel.

01 — Probleme

Le probleme

  • Les professionnels (coiffeurs, coachs, estheticiennes...) gerent leurs RDV par telephone ou papier
  • Les clients veulent reserver en ligne, 24/7
  • Les solutions existantes (Planity, Doctolib) sont couteuses ou trop generiques
  • Les independants n'ont pas d'outil adapte ET abordable

En France, 87% des consommateurs preferent reserver en ligne plutot que par telephone. Pourtant, la majorite des petites structures n'ont toujours pas d'outil de reservation.

02 — Quoi

La solution : CutBook

Plateforme de reservation en ligne adaptable a tout type de profession.

Reservation Booking temps reel Creneaux anti-double-booking. Confirmation instantanee.
Paiement Stripe Checkout Paiement en ligne securise. Webhooks automatiques.
Structure Multi-organisation 1 orga = Owner + Members. Accessible via /@slug.
Calendrier FullCalendar pro Vue semaine/mois interactive. Gestion des dispos.
Engagement Fidelite & Avis Points par RDV. Recompenses auto. Systeme d'avis.
Communication Emails automatiques Confirmation, rappel, annulation via React Email + Resend.
03 — Migration

V1 → V2 : Avant / Apres

Migration
V1 — 2024

Python / Flask "RdVous"

49 fichiers SQLite local Jamais deploye
Stack : Python 3, Flask 3, Jinja2, SQLAlchemy, Flask-WTF, Flask-Login
DB : SQLite (fichier local site.db), migrations Alembic
Auth : Flask-Login + hash Werkzeug, token maison, pas d'OAuth
UI : Jinja2 templates HTML/CSS statiques, FontAwesome 5, aucun composant reutilisable
API : Routes monolithiques (routes.py : 1 312 lignes)
Paiement : Aucun — pas de checkout integre
Email : Flask-Mail + Gmail SMTP, configuration en clair dans le code
Deploy : Local uniquement, SECRET_KEY = 'admin' hardcode
Tests / CI : Aucun — zero couverture automatique
Type-safety : Inexistante, erreurs runtime non detectees
Scope : 1 seul salon («Coiffeur» hard-code), non multi-tenant
V2 — 2026

Next.js 16 "CutBook"

181 fichiers PostgreSQL Neon Deploye Vercel
Stack : Next.js 16 App Router, TypeScript strict, React 19
DB : PostgreSQL (Neon prod / Docker dev), Prisma ORM type-safe
Auth : BetterAuth — email verifie, Google OAuth, sessions securisees
UI : Tailwind v4 + shadcn/ui + Framer Motion, composants reutilisables
API : tRPC end-to-end type-safe + TanStack Query v5
Paiement : Stripe Checkout + Webhooks automatises
Email : React Email + Resend, templates types, env vars securisees
Deploy : Vercel (prod) + CI GitHub Actions, secrets externalises
Tests / CI : Vitest + Testing Library, Husky, ESLint, commitlint
Type-safety : TypeScript strict de bout en bout (DB → API → UI)

↓ 1:1 (Owner)    ↓ 1:1 (Member)

id, name, slug, address, phone, ownerId

↓ 1:N

id, userId, orgId, specialties, bio

id, orgId, name, duration, price

↓ 1:N                ↓ 1:N

id, memberId, date, startTime, endTime

id, clientId, memberId, serviceId, timeSlotId, status, stripePaymentId

Review

id, clientId, memberId, rating, comment

Reward

id, clientId, type, status, expirationDate

05 — Data

Relations & Scalabilite

Relations cles

User ↔ Organization 1:1 (Owner)
User ↔ Member 1:1
Organization ↔ Member 1:N
Organization ↔ Service 1:N
Member ↔ TimeSlot 1:N
TimeSlot ↔ Booking 1:1
User ↔ Booking 1:N (client)
User ↔ Review 1:N
User ↔ Reward 1:N

Scalabilite

Verticale : Neon scale automatiquement (compute auto-suspend, storage elastique). Suffisant pour le MVP.

Horizontale (futur) : le modele organisation permet un sharding par orga. Chaque orga independante, pas de requetes cross-orga.
